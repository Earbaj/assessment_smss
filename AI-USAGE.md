## What I generated with AI

- **Initial code review of the starter file** (`REVIEW.md`): I used Claude
  to help me systematically go through `sms_console.dart` against
  `API-CONTRACT.md` line by line and identify defects (hardcoded API key,
  missing `X-Tenant-Id` header, money handled as `double`, client-side cost
  recomputation instead of trusting the server response, missing `mounted`
  checks, a `recipient` field rendered on a screen the contract says never
  returns one, sensitive data in `print()` logs, and a network call issued
  from inside `build()`).
- **First-pass rebuild of the data layer**: `lib/core/money.dart`,
  `lib/core/models.dart`, `lib/core/api_client.dart`, and
  `lib/core/sms_repository.dart` were generated by AI based on the review
  findings and the API contract, then read through and adjusted by me.

## Where the AI was wrong, or where I didn't trust it

- The AI's first draft of `Money` used `double` internally with rounding
  applied only at display time — I rejected that because the whole point of
  Finding 3 is that arithmetic itself needs to avoid floating point, not
  just formatting. I asked for integer minor-unit arithmetic instead.

## Why this approach

There's no penalty for heavy AI use on this assignment, and no reviewer on
the real job either — so the discipline I wanted to demonstrate here is
knowing which parts I could hand to AI safely (boilerplate parsing code,
doc scaffolding) versus which parts needed my own reasoning because a
mistake there is exactly the kind the starter file's bugs came from in the
first place (money math, tenant scoping, and state transitions a user could
actually hit).
